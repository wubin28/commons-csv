# Java Code Review for commons-csv

Perform a focused code review of the commons-csv project, emphasising correctness of
state-machine logic, boundary conditions, and thread safety.
This skill reflects the current codebase state as of **2026-03-19**.

## Project Layout

| Path | Description |
|------|-------------|
| `src/main/java/org/apache/commons/csv/Lexer.java` | Token parser — multi-char delimiter, escape, trim, EOF detection |
| `src/main/java/org/apache/commons/csv/CSVParser.java` | High-level record parser — comment handling, record numbering |
| `src/main/java/org/apache/commons/csv/CSVRecord.java` | Represents one parsed row — index/name access, null-safety |
| `src/main/java/org/apache/commons/csv/CSVPrinter.java` | Writes CSV — `ReentrantLock`-protected, stream-based printing |
| `src/main/java/org/apache/commons/csv/ExtendedBufferedReader.java` | Byte/char-position tracking, line-number counting, EOF detection |
| `src/main/java/org/apache/commons/csv/CSVFormat.java` | Configuration — delimiter, quote, escape, header, trim options |
| `src/main/java/org/apache/commons/csv/Token.java` | Mutable token value type used during lexing |
| `src/main/java/org/apache/commons/csv/Constants.java` | Named constants for CR, LF, SP, etc. |
| `src/test/java/org/apache/commons/csv/` | JUnit test suite |

## Review Workflow

1. **Identify scope**: single file, a single method, or the full project.
2. **Load the relevant source** with `read_file`.
3. **Apply the checklist** below systematically, top to bottom.
4. **For each finding**: state severity, file and line number, root cause, and a concrete fix with a code diff where helpful.
5. **Consult the Known Hotspots section** first when doing a broad review — these are pre-diagnosed bug patterns unique to this codebase.

## Quick Checklist

### Off-by-One & Boundary Conditions
- [ ] Array/string index bounds: `index < array.length` (strict) vs `index <= array.length` (off-by-one) — must use strict `<` for valid indices
- [ ] `buffer.charAt(length - 1)` vs `buffer.charAt(length)`: `length` is exclusive; accessing `charAt(length)` is out of bounds and throws `StringIndexOutOfBoundsException`
- [ ] Loop termination: `length > 0` vs `length >= 0` in trim loops — stopping at `0` is correct; `>= 0` would under-read by one character
- [ ] Initialisation values: record numbers, counters, and sequence numbers that are documented to start at `1` must be initialised to `1`, not `0`
- [ ] `peek()` vs `read()` when the next character must be consumed as part of a match

### Boolean / Conditional Logic
- [ ] Inverted conditions: `!isLastTokenDelimiter && isEndOfFile(c)` — flipping `&&` to `||` creates a superset trigger that fires at every delimiter *or* every EOF, truncating records
- [ ] Double negation: methods named `isXxx` that are negated at call sites — trace all callers to verify the intended sense
- [ ] Inverted peek guard: `!isQuoteChar(reader.peek())` inside `parseEncapsulatedToken` when the intent is to detect doubled quotes (the guard should be `isQuoteChar`)
- [ ] Null-check placement: guards removed *after* a `getHeaderMapRaw()` call allow NPE when `parser` is `null` (e.g., after Java serialisation)

### State-Machine & Character Tracking
- [ ] Line-number double-count: `\r\n` must increment `lineNumber` exactly once; a wrong `lastChar != CR` or inverted condition causes every CRLF to count as two newlines
- [ ] EOF position inflation: `position++` must be gated on `c != EOF`; incrementing unconditionally on `EOF` returns makes `getPosition()` off by one at end of stream
- [ ] Switch/case return values: in escape-character decode switch statements, each `case` must return the *correct* control character (e.g., `case 'r': return Constants.CR` not `Constants.LF`) — a swap here silently corrupts multi-line field parsing
- [ ] Comment buffer reset: after attaching accumulated comment text to a record, the buffer must be cleared; failing to clear it duplicates the comment across multiple subsequent records

### Multi-Character Delimiter Index Arithmetic
- [ ] `delimiterBuf[i]` vs `delimiter[i + 1]` when peeking ahead for a multi-char delimiter: `delimiterBuf` holds peeked characters starting from index `0`, which corresponds to `delimiter[1]`; using `delimiter[i]` re-reads the first character and causes false mismatches

### Null Safety
- [ ] `CSVRecord.get(String name)` — `getHeaderMapRaw()` returns `null` when the record was deserialised without a live `CSVParser`; accessing the map without a null guard throws NPE instead of a documented `IllegalStateException`
- [ ] `CSVRecord.isSet(String name)` and `putIn(Map<?,?>)` — same null-map risk; each must have its own guard
- [ ] `Optional`-free null returns from internal helpers — prefer explicit null checks at the boundary where the value originates

### Thread Safety & Concurrency
- [ ] `CSVPrinter` uses `ReentrantLock` for `print()`/`println()` but `printRecord(Stream<?>)` calls `printRaw()` for parallel streams (bypassing the lock); ensure the parallel-stream path is either forbidden or properly synchronised
- [ ] `newRecord` flag visibility: if the flag is read and written under different locks (or no lock), it can be observed in an inconsistent state across threads, causing missing or duplicated separators
- [ ] `static` mutable fields — any static cache or counter shared between parser instances must be `volatile` or `AtomicXxx` if read/written from multiple threads

### Exception Handling & Contract Violations
- [ ] Swallowed `IOException`: empty or logging-only `catch (IOException e)` blocks that do not rethrow leave the caller unaware that input was truncated
- [ ] `IllegalArgumentException` vs `IllegalStateException`: use `IllegalArgumentException` for bad input, `IllegalStateException` for object lifecycle violations (e.g., calling methods on a closed parser)
- [ ] Checked exceptions hidden behind unchecked wrappers without preserving the original cause

### Resource Management
- [ ] `CSVParser` and `CSVPrinter` implement `Closeable`; verify that `close()` is idempotent and that the wrapped `Reader`/`Writer` is closed on all exit paths
- [ ] `try-with-resources` used for all `Lexer`, `ExtendedBufferedReader`, and `Reader` instances inside tests and production code
- [ ] `Token` objects reused across iterations — confirm that `reset()` clears *all* mutable fields before reuse

### Java-Specific Pitfalls
- [ ] `int` vs `char` comparison: `read()` returns `int`; comparing with a `char` literal using `==` is safe only after checking `!= -1`; using `(char) read() == SOME_CHAR` loses the EOF signal
- [ ] `String.valueOf((char) -1)` — if EOF is accidentally cast to `char`, it becomes `'\uFFFF'`, not an empty string; guard with `!= EOF` before casting
- [ ] Autoboxing in hot parsing loops: avoid `Character`, `Integer`, or `Boolean` boxing inside the inner loop of `nextToken()`
- [ ] `StringBuilder.setLength(n)` for truncation is correct and preferred; avoid creating a new `StringBuilder` just to drop trailing characters

## Known Hotspots (read first for a full-project review)

These are pre-identified bug patterns unique to this codebase, derived from historical injection-fault analysis.

| # | Hotspot | File | Bug Pattern |
|---|---------|------|------------|
| 1 | `isSet(int)` bounds check | `CSVRecord.java` | `index <= values.length` (off-by-one) should be `index < values.length`; allows a one-past-the-end index to appear valid, causing `get(int)` to throw `ArrayIndexOutOfBoundsException` |
| 2 | `\r\n` line counting | `ExtendedBufferedReader.java` | Inverted `lastChar != CR` guard counts CRLF as two lines; correct guard must suppress the second increment when `lastChar == CR` |
| 3 | EOF token condition | `Lexer.java` (`nextToken`) | `&&` changed to `||` in `!isLastTokenDelimiter && isEndOfFile(c)` fires on every delimiter *or* EOF, truncating the last field of every record |
| 4 | Trim loop index | `Lexer.java` (`trimTrailingSpaces`) | `buffer.charAt(length)` is out of bounds; must use `buffer.charAt(length - 1)` |
| 5 | Escape switch return | `Lexer.java` (`readEscape`) | A swapped return value in `case 'r'` or `case 'n'` corrupts escape decoding for multi-line fields |
| 6 | Quote-doubling guard | `Lexer.java` (`parseEncapsulatedToken`) | `!isQuoteChar(reader.peek())` should be `isQuoteChar(reader.peek())`; inverted guard treats escaped quotes as field terminators |
| 7 | Multi-char delimiter index | `Lexer.java` (`isDelimiter`) | `delimiter[i]` used instead of `delimiter[i + 1]`; repeats the first character and causes false delimiter mismatches |
| 8 | EOF position inflation | `ExtendedBufferedReader.java` (`read`) | Unconditional `position++` after reading EOF inflates `getPosition()` by one |
| 9 | Null header map | `CSVRecord.java` (`get`/`isSet`/`putIn`) | Missing null check after `getHeaderMapRaw()` when `parser == null` (post-deserialisation) causes NPE instead of `IllegalStateException` |
| 10 | Comment buffer not cleared | `CSVParser.java` | `headerComment` buffer not reset after attaching to a record duplicates comment content across consecutive records |
| 11 | Parallel-stream locking | `CSVPrinter.java` | `printRecord(Stream<?>)` bypasses `ReentrantLock` for parallel streams; `newRecord` flag subject to data races |
| 12 | Record number initialisation | `CSVParser.java` (`Builder`) | `recordNumber` initialised to `0` instead of `1` makes the first record have number `0`, violating documented contract |
