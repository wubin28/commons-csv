# Bug 注入报告

## Bug 类型

**差一错误（Off-by-One Error）**

---

## 注入位置

- **文件：** `src/main/java/org/apache/commons/csv/CSVRecord.java`
- **方法：** `isSet(int index)`
- **行号：** 第 259 行

---

## 注入前的代码

```java
public boolean isSet(final int index) {
    return 0 <= index && index < values.length;
}
```

## 注入后的代码

```java
public boolean isSet(final int index) {
    return 0 <= index && index <= values.length;
}
```

**改动摘要：** 将 `index < values.length` 改为 `index <= values.length`，即把严格小于（`<`）改为小于等于（`<=`）。

---

## 该 Bug 有什么危害

`isSet(int index)` 的语义是"第 `index` 个字段是否存在"。`values` 数组是 0-based，合法索引范围是 `[0, values.length - 1]`。

注入后，当 `index == values.length` 时，`isSet` 错误地返回 `true`，误导调用方认为该索引有效。随后若调用 `get(index)`：

```java
public String get(final int i) {
    return values[i];   // values[values.length] → ArrayIndexOutOfBoundsException
}
```

会立即抛出 `ArrayIndexOutOfBoundsException`。这违背了"先用 `isSet` 守卫、再用 `get` 取值"这一惯用模式的安全契约，使防御性代码在边界处反而比直接调用 `get` 更危险。

更隐蔽的危害：`isSet(String name)` 内部也依赖逻辑相似的判断，但使用的是 `< values.length`（未受影响），这会导致按名称访问和按索引访问的边界行为不一致，增加排查难度。

---

## 运行测试代码能否发现这个 Bug

**能被发现。**

执行 `mvn test` 后，`CSVRecordTest.testIsSetInt`（第 219 行）失败：

```
[ERROR] org.apache.commons.csv.CSVRecordTest.testIsSetInt -- FAILURE!
AssertionFailedError: expected: <false> but was: <true>
```

该测试专门断言 `isSet(values.length)` 应返回 `false`，这正是被注入的差一错误触发的边界用例。这说明该项目的测试用例质量较高，**对 `isSet` 的上界越界情况有明确的回归保护**。

测试结果汇总：`Tests run: 924, Failures: 1, Errors: 0`（仅该一条失败）。

---

## 如何修复

将注入的 `<=` 改回 `<`：

```java
// 修复后
public boolean isSet(final int index) {
    return 0 <= index && index < values.length;
}
```

**修复要点：**
- `isSet` 的语义是"是否存在可访问的值"，应与 `get(int)` 的合法索引范围完全一致：`[0, values.length - 1]`。
- 严格使用 `<`（而非 `<=`）确保上界正确排除越界索引。
- 可补充边界单元测试：`assertFalse(record.isSet(record.size()))` 以防回归。
